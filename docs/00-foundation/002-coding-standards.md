# 002 - Coding Standards

**Document ID:** 002

**Document Name:** Coding Standards

**Status:** Draft

---

# 1. Purpose

This document defines the coding standards for the Practice365 codebase.

All code generated by developers or AI assistants must follow these standards.

The purpose is to ensure consistency, maintainability, readability and predictable architecture throughout the project.

---

# 2. General Principles

## CS-001 — Consistency Over Personal Preference

Consistency across the repository is more important than individual coding preferences.

When an existing convention exists, it must be followed.

---

## CS-002 — Readability First

Code should optimize for readability.

Future developers should understand code quickly without requiring additional explanation.

---

## CS-003 — Explicit Code

Prefer explicit code over hidden behavior.

Avoid unnecessary abstractions.

Avoid "magic."

---

## CS-004 — Small Units

Classes, services, functions and components should remain focused on a single responsibility.

Large files should be split into logical units.

---

# 3. TypeScript Standards

- Strict TypeScript mode is mandatory.
- Do not use `any`.
- Prefer interfaces over loose object types.
- Explicit return types are encouraged for exported functions.
- Shared types belong inside the shared packages when reused across modules.

---

# 4. Folder Organization

The repository follows a feature-first architecture.

Example:

apps/api/src/modules/

auth/

users/

clients/

matters/

billing/

Each module owns its own implementation.

Do not create global controller, service or model folders.

---

# 5. Naming Conventions

## Files

Use kebab-case.

Examples

```
user.service.ts

matter.controller.ts

invoice.repository.ts
```

---

## Classes

Use PascalCase.

Example

```
AuthenticationService
```

---

## Variables

Use camelCase.

Example

```
currentUser

invoiceTotal
```

---

## Constants

Use UPPER_SNAKE_CASE.

Example

```
MAX_LOGIN_ATTEMPTS
```

---

## Interfaces

Prefix with "I" only if there is a clear project-wide need.

Otherwise use descriptive names.

Example

```
UserSession

LoginRequest
```

---

## Enums

Use PascalCase.

Enum values should be UPPER_SNAKE_CASE.

---

# 6. Imports

Use absolute imports where practical.

Avoid deep relative imports such as

```
../../../../
```

Shared packages should be imported through configured path aliases.

---

# 7. Functions

Functions should:

- Have one responsibility.
- Be named as verbs.
- Return predictable results.
- Avoid hidden side effects.

---

# 8. Comments

Comments should explain "why."

Do not explain obvious code.

Poor

```ts
// Increment i
i++;
```

Good

```ts
// Preserve historical billing rate for legal audit purposes.
```

---

# 9. Error Handling

Do not swallow exceptions.

Handle expected errors.

Log unexpected errors.

Return standardized API responses.

---

# 10. Logging

Never log:

- Passwords
- JWT Tokens
- Refresh Tokens
- Secrets
- API Keys

Sensitive information must never appear in logs.

---

# 11. Testing

Every module should include:

- Unit Tests
- Integration Tests

Critical business logic should always be tested.

---

# 12. AI Development Rules

AI-generated code must:

- Follow the approved Module PRD.
- Follow these coding standards.
- Avoid inventing business rules.
- Avoid introducing unnecessary dependencies.
- Reuse existing shared utilities where appropriate.
- Preserve consistency with the existing codebase.

---

# 13. Code Review Checklist

Before merging, verify:

- Naming conventions followed.
- Folder structure followed.
- No duplicated logic.
- No unused code.
- No hardcoded secrets.
- TypeScript passes.
- Lint passes.
- Tests pass.
- Module PRD requirements are satisfied.
