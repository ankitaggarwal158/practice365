# PART A — Product Specification

---

# 1. Purpose

The Activity Log & Audit Trail module provides a centralized, immutable history of all significant actions performed within Practice365.

It serves two distinct purposes:

- **Activity Log** – User-friendly activity timeline shown within the application.
- **Audit Trail** – Immutable compliance record for security, legal and operational purposes.

All core modules publish audit events to this module.

---

# 2. Scope

This module includes:

- Activity Timeline
- Audit Logging
- User Activity
- Entity Activity
- Login History
- Search
- Filtering
- Export
- Immutable Audit Records

---

# 3. Out of Scope

This module does not include:

- System Monitoring
- Server Logs
- Infrastructure Logs
- Performance Metrics

Business Module Dependencies

# 5. Business Requirements

## AUDIT-001

Every audit record belongs to exactly one Firm.

---

## AUDIT-002

Audit records are append-only.

Existing audit records can never be modified.

---

## AUDIT-003

Audit records can never be permanently deleted.

---

## AUDIT-004

Every significant business action generates an audit record.

Examples include:

- Login
- Logout
- User Created
- Matter Created
- Matter Updated
- Document Uploaded
- Invoice Issued
- Payment Recorded
- Note Added
- Calendar Event Created
- Signature Completed

---

## AUDIT-005

Activity timelines are generated from audit records.

---

## AUDIT-006

Audit records support filtering by:

- User
- Module
- Entity
- Date Range
- Action Type

---

## AUDIT-007

Audit records may be exported by authorized users.

---

# 8. Data Model

This module follows:

- 003 - Database Conventions

This module owns the following collection.

| Collection | Purpose |
|------------|----------|
| audit_logs | Immutable system audit records |

---

## 8.1 audit_logs

### Purpose

Stores immutable audit events generated by every business module.

This collection serves as the system of record for all significant actions.

---

### Fields

| Field | Type | Description |
|------|------|-------------|
| _id | ObjectId | Primary Key |
| firmId | ObjectId | Firm Reference |
| userId | ObjectId | User performing action |
| module | String | Source Module |
| entityType | String | Entity Type |
| entityId | ObjectId | Entity Reference |
| action | String | Action Performed |
| previousState | Mixed | Previous Values (optional) |
| currentState | Mixed | Updated Values (optional) |
| ipAddress | String | Client IP |
| userAgent | String | Browser Information |
| metadata | Mixed | Additional Context |
| createdAt | Date | Event Timestamp |

---

### Relationships

One Firm

↓

Many Audit Logs

---

One User

↓

Many Audit Logs

---

### Indexes

Create indexes on:

- firmId
- userId
- module
- entityType
- entityId
- action
- createdAt

Composite indexes:

- firmId + createdAt
- firmId + module
- firmId + entityType
- entityId + createdAt

# 12. Security Considerations

This module follows:

- 005 - Security Standards

The following security requirements are specific to the Activity Log & Audit Trail module.

---

## Firm Isolation

Audit records belong to exactly one Firm.

Users may only access audit records belonging to their Firm.

Cross-firm access is prohibited.

---

## Authorization

Only authorized users may access audit records.

Recommended permissions:

- AUDIT_VIEW
- AUDIT_EXPORT

Normal staff should only access activity they are authorized to view.

---

## Immutability

Audit records are immutable.

After creation:

- No updates.
- No deletions.
- No restores.

Corrections require creation of a new audit record.

---

## Data Integrity

Every audit event must include:

- User
- Timestamp
- Module
- Action
- Entity Reference

where applicable.

---

## Export

Audit exports must respect:

- Firm Isolation
- User Permissions
- Date Filters

Exported data must match application data exactly.

---

## Sensitive Information

Audit logs must never store:

- Passwords
- Authentication Tokens
- Secrets
- Encryption Keys

Sensitive values should be masked where required.
